## V1.1 - 2026-06-27
### 优化
- API key 校验启动检查（⭐/⭐⭐⭐⭐/⭐⭐⭐）

启动时先检查 AMAP_KEY 和 OPENAI_API_KEY 是否已设置，没设置就报错退出：


启动 → 检查环境变量
  ├── 缺少 → 打印"请设置 OPENAI_API_KEY 和 AMAP_KEY"
  └── 齐全 → 正常启动
不是高智能优化，但面试官看到"启动校验"会认为你有工程意识。


- 推荐结果持久化展示（⭐/⭐⭐⭐/⭐⭐⭐⭐）

保存行程时不只是存 Markdown，而是在终端显示文件路径后，再打印一行：


已保存到 saved/云南亲子7日游.md
你可随时用 cat saved/云南亲子7日游.md 查看
面试时让面试官跑一下 "--writable 模式保存行程"的 demo，有明确的文件输出，比纯终端输出更显完整。


- 错误分类处理（⭐⭐/⭐⭐⭐⭐/⭐⭐⭐）

现在所有失败统一说"查不到"。改为分类处理：


query_weather 超时 → "天气服务暂时不可用，推荐中不含天气信息"
query_weather 返回空 → "该时段暂无天气预报数据，按历史气候参考推荐"
LLM 提取失败 → "请用更具体的描述重新告诉我"
面试时问"你如何处理 API 失败"时，能给出三级分类而不是一句话，会好很多


- 推荐理由输出（⭐⭐/⭐⭐/⭐⭐⭐⭐⭐）

行程不仅有"每天去哪"，还告诉用户为什么这样推荐：


D1 昆明（滇池+翠湖）
  推荐理由：7月昆明气温20-25°C，适合户外活动。
  滇池是昆明标志性景点，带小孩可以喂海鸥。
  翠湖公园免费，适合到达当天轻松游玩。
这个功能代码简单（让 LLM 在 dailyPlan 里加一个 reason 字段），但面试演示效果非常好——评委看到的不只是结构化的行程，而是可解释的推荐。


- 兜底推荐策略（⭐⭐⭐/⭐⭐⭐⭐/⭐⭐⭐）

所有工具都失败时怎么办？不要直接报错，而是：


天气查不到、景点查不到 → "根据我的知识，大理7月气候温和，
推荐以下经典行程（以下为通用推荐，非实时数据）..."
面试时可以表述为："我们设计了 graceful degradation 策略，确保即所有工具都失败，用户也能拿到有意义的回复。"


- 工具调用次数限制（⭐/⭐⭐⭐/⭐⭐⭐⭐）

加一个最大工具轮数限制（如 8 轮），超过就输出已有结果并提示"已查询多轮，建议基于已有信息规划"。防止 Agent 陷入无限工具调用循环。
面试时提到这个说明你理解 tool_use 的工程陷阱。
